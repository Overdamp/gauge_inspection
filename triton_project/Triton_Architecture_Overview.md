## 1. ภาพรวมของระบบเชิงลึก (Deep-Dive System Overview)

ในอดีต การพัฒนาซอฟต์แวร์ประมวลผล AI มักทำในรูปแบบ **Monolith (แอปพลิเคชันก้อนเดียว)** เช่น โค้ด Python (ด้วย Flask หรือ FastAPI) โหลดโมเดล AI ขึ้นมาไว้ในหน่วยความจำ แล้วรันบนเว็บเซิร์ฟเวอร์เดียว ซึ่งแนวทางนี้มี **ข้อจำกัดร้ายแรง (Critical Architectural Flaws)** เมื่อขยายสู่ระดับอุตสาหกรรม:

* **VRAM Memory Leaks:** โมเดลบางประเภทคายหน่วยความจำใน GPU ไม่สมบูรณ์ ทำให้เครื่องเซิร์ฟเวอร์ค้างหลังรันไปนานๆ
* **GIL (Global Interpreter Lock) Block in Python:** เว็บ API ไม่สามารถประมวลผลคู่ขนานแบบ High-Throughput ได้อย่างแท้จริง เพราะ Python รันได้ทีละ Thread เมื่อยิงรูปภาพเข้ามาพร้อมๆ กันจะทำให้ระบบค้างสะสม
* **Single Framework Lock-in:** การรัน PyTorch คู่กับ TensorRT หรือ TensorFlow ในแอปพลิเคชันเดียวกันทำได้ยากและเกิดปัญหาไลบรารีชนกัน (Dependency Conflicts)

ด้วยเหตุนี้ สถาปัตยกรรมนี้จึงเปลี่ยนมาใช้ **Microservices (การแยกบริการตามหน้าที่)** อย่างเด็ดขาด โดยแยกภาระงานออกเป็น 2 ชั้นประมวลผล:

```
┌──────────────────────────────────────────────────────────────┐
│                  Client Layer (Robots, CCTVs)                │
└──────────────────────────────┬───────────────────────────────┘
                               │ HTTP / JSON (Payload A/C)
┌──────────────────────────────▼───────────────────────────────┐
│        AI Inference Proxy (CPU-bound Layer: API Gateway)     │
│ - รับ JSON, ตรวจสอบสิทธิ์ (Security API Keys)                 │
│ - ดาวน์โหลดภาพและจัดเตรียมข้อมูล (JPEG Decode, Resize, CPU)  │
│ - แปลงผลลัพธ์เป็นมาตรฐานสากล (JSON Version 8 Schema)          │
└──────────────────────────────┬───────────────────────────────┘
                               │ gRPC / Binary Tensors (Payload B)
┌──────────────────────────────▼───────────────────────────────┐
│         NVIDIA Triton Server (GPU-bound Layer: AI Engine)    │
│ - จัดคิวงานด้วย Dynamic Batching Scheduler                   │
│ - รันโมเดลหลายตัวขนานกันแบบ Concurrent ใน VRAM               │
│ - ส่งการคำนวณเข้าแกนสมอง TensorRT/PyTorch บนการ์ดจอ (GPU)    │
└──────────────────────────────────────────────────────────────┘
```

> [!TIP]
> **Separation of Concerns:** การแยก Proxy (CPU-bound) ออกจาก Triton (GPU-bound) ช่วยให้เซิร์ฟเวอร์สามารถใช้การ์ดจอราคาแพง (เช่น NVIDIA A6000) คำนวณ AI ได้ต่อเนื่องเกือบ 100% ของความสามารถ โดยไม่ต้องเสียเวลา CPU ไปกับภารกิจย่อยอื่นๆ

## 2. รูปแบบการสื่อสารและการไหลของข้อมูลแบบเจาะจง (Payload Lifecycle & Network Protocols)

ระบบกำหนดให้ข้อมูลที่จะส่งไปตรวจวัดและอ่านผลค่าทางวิศวกรรมต่างๆ (เช่น วัดเข็มเกจ Task 1, ตรวจจับแก๊สรั่ว Task 4) ไหลผ่านตัวแปร 3 รูปแบบหลัก เพื่อควบคุมคุณภาพและประสิทธิภาพ

### 2.1 โครงสร้างข้อมูลในแต่ละระยะ (Payload Definitions)

#### 1) Payload A (Client Input - JSON Schema)

เป็นข้อมูลที่มีน้ำหนักเบา ส่งมาจากหุ่นยนต์หรือระบบควบคุมกลาง (CCR) เพื่อลดภาระการรับส่งข้อมูลของเครือข่ายไร้สาย (Wi-Fi/5G) ในโรงงานปฏิบัติงาน

```json
{
  "task_id": "job_analog_gauge_99",
  "task_type": 1,
  "image_source": {
    "type": "s3",
    "url": "s3://pttep-storage/patrol_images/2026-05-18/gauge_001.jpg"
  },
  "parameters": {
    "gauge_type": "pressure",
    "expected_min": 0.0,
    "expected_max": 10.0
  }
}
```

#### 2) Payload B (Internal Data - Binary Tensors)

ตัว Proxy ทำการแปลง Payload A โดยการดาวน์โหลดรูปภาพ `.jpg` จาก URL แปลงสัญญาณภาพเป็นอาเรย์ตัวเลขสามมิติ (Height x Width x Channels) หรือที่เรียกว่า **Tensor** และแนบพารามิเตอร์อื่นๆ ในรูปแบบตัวเลขดิบ เพื่อประหยัดพื้นที่ข้อมูลแล้วยิงเข้า Triton ผ่านพอร์ต **gRPC** ในรูปของไบนารีส่งตรงเข้า VRAM ของ GPU

* **Input Image Tensor:** `[1, 3, 640, 640]` (Float32)
* **Input Parameters:** `[0.0, 10.0]` (Float32)

#### 3) Payload C (System Output - Standard JSON Schema)

ผลลัพธ์จาก AI จะถูกแปลงโดย Proxy กลับไปเป็น JSON ที่สวยงามและมีโครงสร้างแบนราบ (Flat JSON Version 8) ซึ่งหุ่นยนต์สามารถนำไปเก็บลงฐานข้อมูลหรือส่งแจ้งเตือนต่อได้ทันที

```json
{
  "task_id": "job_analog_gauge_99",
  "status": "success",
  "timestamp": "2026-05-18T11:55:00Z",
  "task_type": 1,
  "outputs": [
    {
      "value": 6.8,
      "unit": "bar",
      "confidence": 0.984,
      "abnormality_detected": false,
      "bounding_box": [120, 80, 340, 300]
    }
  ]
}
```

### 2.2 โปรโตคอลของเน็ตเวิร์ก (Network Protocols Comparison)

ในการสื่อสาร เราใช้สองโปรโตคอลที่แตกต่างกันอย่างจงใจเพื่อรีดประสิทธิภาพ:

| คุณสมบัติ | HTTP/REST (ใช้เชื่อมต่อภายนอก) | gRPC (ใช้เชื่อมต่อภายใน) |
| :--- | :--- | :--- |
| **การใช้งาน** | หุ่นยนต์ / CCR ยิงหา Inference Proxy | Inference Proxy ยิงส่งรูปเข้า Triton Server |
| **ความเร็ว** | ปานกลาง (มี Overhead จากหัว JSON) | เร็วสูงสุดขีด (Binary Serialization) |
| **การแลกเปลี่ยนข้อมูล** | Text/JSON (คนอ่านเข้าใจง่าย) | Protocol Buffers (คอมพิวเตอร์แปลงเร็วสุด) |
| **การเชื่อมต่อ** | สั้น (เปิดและปิดพอร์ตบ่อยครั้ง) | HTTP/2 (เปิดพอร์ตค้างไว้เพื่อส่งสตรีมต่อเนื่อง) |

## 3. การวิเคราะห์ความจำเป็นของ Inference Proxy (Proxy vs. Direct Triton Access)

คำถามเชิงสถาปัตยกรรมที่สำคัญที่สุดคือ: **"เราสามารถให้หุ่นยนต์ยิงตรงหา Triton Server เลยได้ไหม?"**
คำตอบคือ **ได้ทางทฤษฎี แต่ห้ามทำเด็ดขาดในทางปฏิบัติเชิงอุตสาหกรรม (Anti-pattern in Production)** เนื่องจากสาเหตุดังนี้:

### 3.1 คอขวดการดาวน์โหลดและถอดรหัสรูปภาพ (Image Decoding Bottleneck)

รูปภาพที่ส่งมาจากหุ่นยนต์มักเป็นไฟล์บีบอัดเช่น `.jpg` หรือ `.png` การจะนำภาพเหล่านี้เข้าประมวลผล AI จะต้องผ่านขั้นตอน:

1. ดาวน์โหลดไฟล์ภาพผ่าน Network (เช่น จาก Cloud Storage S3)
2. ถอดรหัสรูปภาพ (JPEG Decompression) ให้เป็นพิกเซลดิบ (Raw Pixels/Numpy Array)
3. ปรับขนาดภาพ (Resizing) และการแปลงสเกลสี (Normalization)

ขั้นตอนทั้งหมดด้านบนนี้ใช้พลังงานของ **CPU** และระบบไฟล์ (I/O) เป็นหลัก หากเราส่งไฟล์ภาพบีบอัดนี้ไปให้ Triton Server โดยตรง ตัว Triton จะต้องใช้ CPU ของตัวเองทำขั้นตอนเหล่านี้ ซึ่งทำให้การ์ดจอ (GPU) ต้องนั่งว่างงานเพื่อรอข้อมูล (GPU Starvation) การแยก Proxy ออกมาช่วยทำหน้าที่นี้บน CPU ของอีกชุดระบบ ทำให้ Triton ได้รับข้อมูลดิบและรันโมเดลบน GPU ได้ทันทีโดยไม่เสียเวลารอคอย

### 3.2 ความซับซ้อนของอินพุต/เอาต์พุต (Complexity of Tensors)

Triton คาดหวังการคุยด้วย Tensors ดิบ ซึ่งเป็นอาเรย์ไบนารีที่จัดพอร์ตเรียงตามมิติตัวเลข หากหุ่นยนต์ต้องทำสิ่งนี้เอง:

* ซอฟต์แวร์บนหุ่นยนต์จะต้องลงไลบรารีขนาดใหญ่ เช่น `numpy`, `tritonclient`, `opencv`
* สเปคคอมพิวเตอร์บนหุ่นยนต์ (Edge Compute) จะแบกรับภาระหนักเกินไปจนแบตเตอรี่หมดไว
* การมี Proxy ช่วยให้หุ่นยนต์ทำตัวเบาบางที่สุด (Thin Client) โดยคุยแค่ JSON ธรรมดาๆ เท่านั้น

## 4. การจัดการคลังโมเดลผ่าน Docker Volume Mounting และกลไกโหลดแบบ Dynamic

ระบบใช้ประโยชน์จาก **Docker Volume Mounting** เพื่อเชื่อมไฟล์จากไดเรกทอรีถาวรบนเซิร์ฟเวอร์เครื่องหลัก (Host System) เข้าไปหาแอปพลิเคชัน Triton ที่รันอยู่ในตู้ปิดคอนเทนเนอร์ (Docker Container)

```bash
docker run --gpus all \
  -d \
  --name tritonserver_prod \
  --net=host \
  --shm-size=10g \
  -v /home/luke/gauge_inspection/triton_project/model_repository:/models \
  nvcr.io/nvidia/tritonserver:23.08-py3 \
  tritonserver \
  --model-repository=/models \
  --model-control-mode=poll \
  --repository-poll-secs=10
```

### เจาะลึกออปชันคำสั่งวิศวกรรมระบบ (System Parameters Deep-Dive)

* `--gpus all`: มอบอำนาจให้ Triton สามารถดึงทรัพยากรการ์ดจอทุกใบในเครื่องมาประมวลผลร่วมกัน
* `--shm-size=10g`: ตั้งค่าขนาด **Shared Memory (หน่วยความจำร่วมระหว่างระบบปฏิบัติการ)** ให้กว้างถึง 10 GB เพื่อช่วยให้ Proxy และ Triton สามารถส่งข้อมูล Tensors ขนาดใหญ่ผ่านหน่วยความจำ RAM ตรงโดยไม่ต้องผ่านเน็ตเวิร์กการ์ด (ขจัด Network Overhead)
* `--model-control-mode=poll`: สั่งให้ Triton คอยสแกนโฟลเดอร์โมเดลเรื่อยๆ
* `--repository-poll-secs=10`: กำหนดให้ Triton สแกนหาโมเดลใหม่ทุกๆ 10 วินาที หากตรวจพบโมเดลเวอร์ชันใหม่ ตัวระบบจะสั่งให้ VRAM สลับไปโหลดโมเดลตัวใหม่ทันทีและปลดโมเดลตัวเก่าออก โดยไม่ส่งผลกระทบต่องานประมวลผลปัจจุบัน (Zero Downtime Hot Swap)

### 4.2 วงจรการโหลดโมเดลและรายละเอียดของ Model Control API (Dynamic Hot-Swapping Modes)

เพื่อรีดประสิทธิภาพการทำงานสูงสุด Triton จะไม่ทำการอ่านไฟล์จากฮาร์ดดิสก์ใหม่ทุกครั้งที่มีรูปภาพส่งเข้ามาประมวลผล แต่จะทำการจองพื้นที่และดึงโมเดลขึ้นหน่วยความจำแบบแคชถาวร (In-Memory Caching) โดยมีวงจรชีวิตของระบบ (Model Lifecycle) และทางเลือกโหมดควบคุมการโหลดดังนี้:

#### 1) วงจรการโหลดข้อมูลในระดับ Python Backend Code
เมื่อเราเขียนโมเดลควบคุมตรรกะในไฟล์ `model.py` Triton จะประมวลผลด้วยวงจรชีวิต 3 สถานะ:

*   **`initialize(self, args)` [รันครั้งเดียวตอนเริ่ม]:** Triton จะทำงานเมื่อโมเดลถูกเปิดใช้งานครั้งแรกเท่านั้น ในสเต็ปนี้โค้ดของเราจะไปดึงไฟล์ตั้งค่ากลาง [config.yaml](file:///home/luke/gauge_inspection/configs/config.yaml) และทำการโหลดไฟล์เวทน้ำหนัก เช่น `best_segment_v2.pt` ขึ้นจองพื้นที่บน GPU VRAM จนสำเร็จ
*   **`execute(self, requests)` [รันทุกครั้งที่มีงานเข้ามา]:** เมื่อหุ่นยนต์ยิงภาพเข้ามา ตัวประมวลผลจะหยิบส่งไปประมวลผลผ่านตัวแปรโมเดลบน GPU VRAM ทันที โดยไม่มีการเรียกอ่านดิสก์ซ้ำ จึงทำความเร็วได้ในระดับมิลลิวินาที
*   **`finalize(self)` [รันครั้งเดียวตอนถอนโมเดล]:** ทำการปลดปล่อยตัวแปร AI ทั้งหมดออกจาก VRAM และ RAM เพื่อคืนคืนพื้นที่ความจุให้ระบบอื่นนำไปใช้งานต่อ

#### 2) เจาะลึกโหมดควบคุมของ Model Control API
ในการกำหนดพารามิเตอร์ `--model-control-mode` ตอนรันเซิร์ฟเวอร์ Triton มี 3 ทางเลือกที่ส่งผลต่อการเปลี่ยนผ่านอัปเกรดระบบ:

| โหมดควบคุม (Model Control Mode) | ลักษณะการทำงานและพฤติกรรมของ Triton | เคสการใช้งานที่เหมาะสม (Best Use Cases) |
| :--- | :--- | :--- |
| **`none` (โหมดเริ่มต้นระบบปิด)** | สแกนหาโมเดลทั้งหมดในคลังเฉพาะตอนเปิดระบบครั้งแรก โหลดโมเดลทั้งหมดขึ้น VRAM และปิดความสามารถในการเปลี่ยนแปลงข้อมูลผ่านเน็ตเวิร์กอย่างถาวร | **Development & Strict Security:** เหมาะกับช่วงทดสอบโครงสร้างระบบ หรือเซิร์ฟเวอร์ที่ต้องการเสถียรภาพสูงสุดโดยไม่มีใครมาลอบแก้น้ำหนักกลางอากาศ |
| **`poll` (โหมดสแกนอัตโนมัติ)** | คอยใช้ Thread คอยดึงข้อมูลความเคลื่อนไหวจากไดเรกทอรี `model_repository/` เสมอ หากตรวจพบการเปลี่ยนรูปไฟล์ `.pt` หรือ config.pbtxt จะทำลาย VRAM ตัวเก่าและโหลดใหม่โดยอัตโนมัติ | **Continuous Integration:** เหมาะสำหรับเซิร์ฟเวอร์ระดับการพัฒนาร่วมกันที่ทีมนักพัฒนาต้องการผลักดันโมเดลใหม่ ๆ ขึ้นไปรันเช็คผลทันทีโดยไม่ต้องสัมผัส CLI |
| **`explicit` (โหมดอัปเกรดแบบไร้การสะดุด)** | Triton จะไม่โหลดโมเดลใด ๆ ขึ้น GPU เลยตอนเปิดเครื่อง แต่จะรอการยิงสั่งคำสั่งควบคุมภายนอกผ่าน HTTP POST/gRPC API เพื่อสั่งโหลดหรือลบรายโมเดลอย่างเจาะจง | **Multi-Robot Production Fleet:** เหมาะสำหรับระบบที่มีหุ่นยนต์หลายสิบตัวทำงานอยู่ เพื่ออัปเกรดเวอร์ชันโมเดล A โดยไม่ส่งผลกระทบต่อหุ่นยนต์ที่ใช้บริการโมเดล B |

#### 3) สถานะระบบและการควบคุมของเราเตอร์ในปัจจุบัน
*   **สถานะปัจจุบัน:** ตู้คอนเทนเนอร์ `triton_bls` กำลังทำงานภายใต้โหมด **`none` (ตั้งต้น)** เพื่อความเรียบง่ายและปลอดภัยสูงสุดในช่วงพัฒนา ทำให้เมื่อเราอัปเกรดสถาปัตยกรรมคลังโมเดลเสร็จสิ้น เราจะทำการสั่ง `docker restart triton_bls` เพื่อสแกนล้างใหม่ทั้ง 16 โมเดล
*   **เป้าหมายการเปลี่ยนผ่านระดับ Production:** เมื่อโรงงานเริ่มมีหุ่นยนต์ลาดตระเวนปฏิบัติงานจริงหลายสิบตัว การเปิดใช้งานโหมด **`explicit`** ร่วมกับการเขียนระบบหลังบ้านสั่ง `POST /v2/repository/models/<model_name>/load` จะทำให้เราสามารถทำ **Zero-Downtime Live Updates** ปล่อยอัปเดตโมเดลเวอร์ชันใหม่ได้ตลอดเวลา 24 ชั่วโมงโดยไม่มีการตัดสายสัญญาณหุ่นยนต์เลยครับ!

## 5. การจัดโครงสร้างคลังโมเดลระดับสเกลใหญ่ (Super Scale Model Repository Directory Structure)

เนื่องจาก Triton มี **ข้อจำกัดเชิงโครงสร้างที่เข้มงวด (Strict Layout Constraints)** โครงสร้างโมเดลจึงต้องเป็นแบบแบนราบแนวระดับเดียวกัน (Flat Directory Structure) ห้ามซ้อนโฟลเดอร์ลึกเกินกำหนด ดังโครงสร้างอ้างอิงจริงดังนี้:

```
model_repository/
├── task_analog_gauge_router/       <-- โมเดลหลักประเภท Router (BLS)
│   ├── config.pbtxt                <-- ไฟล์ตั้งค่าพอร์ตและการไหลของงาน
│   └── 1/
│       └── model.py                <-- สคริปต์ควบคุมตรรกะการเรียกใช้โมเดลย่อย
│
├── task_gas_detection_router/      <-- โมเดลหลักประเภท Router ประจำงานตรวจจับแก๊ส
│   ├── config.pbtxt
│   └── 1/
│       └── model.py
│
├── shared_yolo_detector/           <-- โมเดลย่อยส่วนกลางทำหน้าที่หาตำแหน่งวัตถุ/เกจ
│   ├── config.pbtxt                <-- ระบุ Backend เป็น TensorRT (.engine)
│   └── 1/
│       └── model.engine            <-- โมเดลคอมไพล์แล้วโดย TensorRT เพื่อความเร็วสูงสุด
│
└── shared_easyocr_engine/          <-- โมเดลย่อยส่วนกลางสำหรับอ่านตัวเลข
    ├── config.pbtxt                <-- ระบุ Backend เป็น PyTorch (.pt)
    └── 1/
        └── model.pt
```

### ทำไมถึงเรียกโครงสร้างนี้ว่าประหยัดการ์ดจอระดับสเกลใหญ่?

สมมติว่าเรามี 5 งานที่ต้องการใช้การตรวจจับวัตถุ (YOLO) เพื่อหาเป้าหมาย หากเราจัดโครงสร้างแบบเก่า (ซ้อนโมเดลแยกกันในแต่ละงาน) ตัวโมเดล YOLO จะถูกโหลดขึ้น VRAM ของการ์ดจอถึง 5 ครั้ง (สมมติโมเดลละ 2GB จะสูญเสีย VRAM ไปถึง 10GB!)

การจัดเก็บแบบ **Shared Backbone** อนุญาตให้เราเปิดโมเดล `shared_yolo_detector` ในความจำเพียงครั้งเดียว (กินเนื้อที่ 2GB เท่าเดิม) แต่ทั้ง `task_analog_gauge_router` และ `task_gas_detection_router` สามารถวิ่งเข้ามารุมเรียกใช้งานโมเดลแชร์ตัวเดียวกันนี้ได้พร้อมๆ กันอย่างปลอดภัย

### 5.2 การเลือกระหว่าง Ensemble Model (3 ตู้แยก) กับ Hybrid Python Wrapper (ตู้เดี่ยวผสม)

เพื่อเตรียมความพร้อมในการสเกลระบบ และจัดโครงสร้างการทำ Pre/Post-processing ของตัวแชร์ทรัพยากร (`shared_*`) เราสามารถเลือกระหว่างสองแนวทางการออกแบบเชิงลึกดังนี้:

#### 1) แนวทางที่ 1: ตู้เดี่ยวผสม (Hybrid Python Wrapper) - [แนวปฏิบัติหลักที่เลือกใช้ในระบบนี้ 🌟]
ในแนวทางนี้ ตัวโมเดลประมวลผลย่อย (เช่น `shared_yolo_detector`) จะบรรจุเฉพาะไฟล์เวทโมเดลดิบที่เป็นมาตรฐาน เช่น `.onnx` หรือ TensorRT `.plan/.engine` (ไม่มีไฟล์สคริปต์ Python ในโฟลเดอร์รหัสรุ่น)
* **กลไกการทำงาน:** 
  1. ตัวควบคุมตรรกะรายภารกิจ (`task_*_router` ที่รันด้วย Python BLS) จะทำหน้าที่ทั้งในการทำ Preprocessing (เช่น การลด/เพิ่มขนาดรูปภาพด้วย OpenCV, ปรับสเกลช่วงสีเป็นทศนิยม และสลับแกนสี)
  2. โอนก้อนทศนิยม `float32` ที่นวดสำเร็จแล้วตรงเข้าไปที่โมเดลเวทดิบ (`shared_yolo_detector`) เพื่อคำนวณผ่าน GPU ด้วยประสิทธิภาพ C++ สูงสุด
  3. รับค่าตัวเลขพิกัดทำนายดิบกลับมา แล้วดำเนินการกรองตำแหน่งกรอบ (Postprocessing/NMS) ภายในโค้ดเราเตอร์หลัก
* **ข้อดี:** ง่ายในการพัฒนาและบำรุงรักษา โครงสร้างคลังโมเดลราบเรียบ (Flat) ดูแลรักษาง่ายขึ้น 4 เท่า และสามารถแก้ไข/ปรับเปลี่ยนพฤติกรรมภาพและตรรกะได้ทันทีโดยแก้ในไฟล์เราเตอร์ชุดเดียว
* **ความเร็วประมวลผล:** ดีเยี่ยมระดับมิลลิวินาที คุ้มค่าและตอบสนองไวกว่า Monolith แบบเดิมมาก

#### 2) แนวทางที่ 2: ระบบสายพานเต็มรูปแบบ (Triton Ensemble Pipeline)
แยกการประมวลผลของ Backbone แต่ละตัวออกเป็น 3 โฟลเดอร์อิสระในคลังโมเดล:
1. `shared_yolo_preprocess` (Python Backend) — ทำหน้าที่นวดภาพดิบ `uint8` ให้เป็นทศนิยม `float32`
2. `shared_yolo_onnx_engine` (ONNXRuntime Backend) — บรรจุเวท `model.onnx` ยิงคำนวณการ์ดจอเพียว ๆ
3. `shared_yolo_postprocess` (Python Backend) — รับพิกัดดิบมากรองกรอบ (NMS)
และสร้างโฟลเดอร์หลักขึ้นมาเพื่อมัดรวมสามชิ้นนี้เชื่อมต่อแรมตรงกันโดยอัตโนมัติ (`platform: "ensemble"`)
* **ข้อดี:** ความเร็วสูงสุดเท่าที่เป็นไปได้ ปราศจากพันธนาการ Python GIL ในสเต็ปโมเดลหลัก
* **ข้อเสีย:** ดูแลรักษายากมากและหัวจะปวด มีความซับซ้อนของไฟล์ `config.pbtxt` มหาศาล หากโยงพอร์ตผิดเพียงอักษรเดียว Triton จะเปิดไม่ขึ้นและดีบั๊กได้ยากมาก

**สรุปข้อตกลงสถาปัตยกรรม:** สำหรับระบบ Gauge Inspection ในเวอร์ชันปัจจุบันและแผน V2 เราตกลงเลือกแนวทาง **"ตู้เดี่ยวผสม (Hybrid Python Wrapper)"** เนื่องจากมอบเสถียรภาพและประสิทธิภาพที่สูงยิ่งยวด ในขณะเดียวกันก็รักษาความเรียบง่ายและเพิ่มขีดความสามารถในการปรับปรุงตรรกะความยืดหยุ่นให้ทีมพัฒนาระบบได้อย่างรวดเร็วครับ!

## 6. เจาะลึกโครงสร้างวิศวกรรมภายในของ NVIDIA Triton (Internal Architecture Breakdown)

นี่คือการแกะกล่องอธิบายสถาปัตยกรรมภายในตามภาพพิมพ์เขียวอย่างเป็นทางการของ NVIDIA เพื่อให้เข้าใจการไหลเวียนข้อมูลภายในตัวเซิร์ฟเวอร์ AI:

```
                                  Client Request
                                        │ (gRPC/HTTP)
  ┌─────────────────────────────────────▼─────────────────────────────────────┐
  │ 1. API & Control Gateway (พอร์ตทางเข้า)                                    │
  └─────────────────────────────────────┬─────────────────────────────────────┘
                                        │
  ┌─────────────────────────────────────▼─────────────────────────────────────┐
  │ 2. Dynamic Batching & Queue Scheduler (ระบบคัดกรองคิวและมัดรวมข้อมูล)        │
  └─────────────────────────────────────┬─────────────────────────────────────┘
                                        │
  ┌─────────────────────────────────────▼─────────────────────────────────────┐
  │ 3. Memory Manager (จัดการ Pinned Memory & Shared VRAM)                    │
  └─────────────────────────────────────┬─────────────────────────────────────┘
                                        │
  ┌─────────────────────────────────────▼─────────────────────────────────────┐
  │ 4. Framework Backends Execution (แกนประมวลผลเฉพาะของแต่ละโมเดล AI)          │
  │    ┌──────────────┐  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐ │

  │    └──────────────┘  └──────────────┘  └──────────────┘  └──────────────┘ │
  └─────────────────────────────────────┬─────────────────────────────────────┘
                                        │
  ┌─────────────────────────────────────▼─────────────────────────────────────┐
  │ 5. Hardware Compute Layer (ชิปการ์ดจอ GPU / แกนประมวลผล CPU)              │
  └───────────────────────────────────────────────────────────────────────────┘
```

### 6.1 รายละเอียดการทำงานของแต่ละชั้นประมวลผล (Core Component Breakdown)

* **1. API & Control Gateway (ด่านแรก):**
    รองรับทั้งโปรโตคอล HTTP/gRPC และช่องทาง C API โดยด่านควบคุมนี้ทำหน้าที่กรองความปลอดภัยของก้อนไบนารี แปลงการร้องขอให้อยู่ในโครงสร้าง Tensor มาตรฐาน และตรวจสอบสุขภาพความถูกต้องของคำขอเพื่อเตรียมนำไปเข้าคิว
* **2. Dynamic Batching & Queue Scheduler (สมองจัดการคิว):**
    เมื่อมีคำร้องขอเข้ามาในเวลาเดียวกัน ระบบจะกระจายตัวส่งข้อมูลเข้าไปที่ **Per-Model Queue (คิวเฉพาะรายโมเดล)** ป้องกันไม่ให้การประมวลผลโมเดลที่ช้าไปอุดคิวโมเดลที่ทำงานเร็ว จากนั้น Scheduler จะสแกนคิวและจับมัดรวมข้อมูล (Batching) ตามเกณฑ์เวลาที่เราตั้งค่าไว้ในไฟล์ `config.pbtxt` เพื่อป้อนข้อมูลให้ฮาร์ดแวร์ทำงานได้คุ้มค่าที่สุด
* **3. Memory Manager (การควบคุมหน่วยความจำชั้นสูง):**
    จัดการใช้เทคนิค **Pinned Memory (การล็อกหน่วยความจำระบบ)** ป้องกันไม่ให้ระบบปฏิบัติการปัดข้อมูล Tensors ตกไปอยู่ใน Harddisk ชั่วคราว (Swap Area) ส่งผลให้การถ่ายโอนข้อมูลจาก RAM บนเมนบอร์ด ไปยัง VRAM บนการ์ดจอทำได้ไวที่สุดผ่านบัสความเร็วสูง PCIe
* **4. Framework Backends Execution (เครื่องยนต์ประมวลผลโมเดล):**
    ตัวรันระบบประมวลผล AI ของแต่ละค่าย เช่น หากเป็นโมเดลที่คอมไพล์มาเพื่อความเร็วสูงสุด Triton จะดึงแกน **TensorRT Backend** มาคำนวณ หากเป็นโมเดลตรรกะซับซ้อนที่เพิ่งเขียนเสร็จใหม่ๆ ก็จะเรียกใช้ **Python/PyTorch Backend** โดยตัวประมวลผลย่อยทั้งหมดจะถูกรันขนานกันอย่างอิสระ
* **5. Hardware Compute Layer (ฐานล่างสุดของฮาร์ดแวร์):**
    การส่งชุดคำสั่งตัวเลขดิบที่ผ่านการกรองและจัด Batching เรียบร้อยแล้วเข้าไปยังแกนสมอง Tensor Cores ของการ์ดจอ NVIDIA เพื่อทำการคำนวณเลขทศนิยมล้านล้านสเต็ปในเสี้ยววินาที

## 7. 10 ฟีเจอร์หลักของ NVIDIA Triton เชิงสถาปัตยกรรม (Detailed Technical Features Analysis)

นี่คือการวิเคราะห์และชี้แจงประโยชน์การใช้งานระดับองค์กรของทั้ง 10 คุณสมบัติหลักที่ทำให้ NVIDIA Triton โดดเด่นเหนือกว่าระบบเสิร์ฟโมเดลทั่วไปในท้องตลาด:

### 1) Supports Multiple Deep Learning Frameworks (การยุบรวมระบบประมวลผลเข้าเป็นหนึ่งเดียว)

* **เจาะลึกการทำงาน:** Triton รันแกนตัวแปลงโมเดลหลายตัวไปพร้อมกัน เช่น ตัวถอดรหัสของ PyTorch (.pt), ตัวเปิดโมเดล ONNX (.onnx) และตัวประมวลผลกราฟฟิกของ TensorRT (.engine)
* **เคสการใช้งานจริง:** ในระบบตรวจวัดเกจ เราใช้โมเดลระบุพิกัดแบบ TensorRT (เพื่อความเร็ว) ส่งต่อข้อความให้โมเดลอ่านอักษรที่เป็น PyTorch (เนื่องจากโค้ดซับซ้อน) โดยสามารถรันควบคู่กันได้ทันที ไม่ต้องรันแยกเว็บเซิร์ฟเวอร์ให้สิ้นเปลืองพลังงานเซิร์ฟเวอร์

### 2) Supports Classic Machine Learning (รองรับอัลกอริทึมคณิตศาสตร์ดั้งเดิม)

* **เจาะลึกการทำงาน:** ใช้ไลบรารี **NVIDIA FIL (Forest Inference Library)** รันโมเดลแบบโครงสร้างต้นไม้ตัดสินใจ (Tree-based Models เช่น Random Forest, XGBoost) บน GPU
* **เคสการใช้งานจริง:** นอกจากวิเคราะห์กล้องอ่านเกจแล้ว ลูกค้าอาจมีข้อมูลเซ็นเซอร์อุณหภูมิและความดันส่งเข้ามาในคราวเดียวกันเพื่อหาโอกาสเกิดเครื่องจักรล้มเหลว (Predictive Maintenance) โดยใช้โมเดล XGBoost ซึ่งสามารถรันผ่าน Triton เพื่อทำนายผลร่วมกับโมเดลสายภาพได้ทันที

### 3) Concurrent Model Execution (การจัดสรรพลังประมวลผลระดับวินาที)

* **เจาะลึกการทำงาน:** การตั้งค่าเปิดโมเดลจำลองตัวเดียวกันขึ้นมาหลายชุด (Instance Groups) เพื่อรันคู่ขนานกันผ่านการควบคุม CUDA Streams
* **เคสการใช้งานจริง:** เมื่อมีกล้องและหุ่นยนต์ 10 ตัวถ่ายภาพส่งเข้ามาในเวลาเดียวกัน Triton จะสลับส่งงานเข้า GPU เพื่อทำงานคู่ขนานกันได้อย่างสมบูรณ์แบบ ทำให้การ์ดจอทำงานได้เต็มประสิทธิภาพและขจัดปัญหารอคิวนาน

### 4) Dynamic Batching (ตัวจัดระเบียบตารางและมัดรวมขนาดยักษ์)

* **เจาะลึกการทำงาน:** รวบคำขอที่เข้ามาพร้อมๆ กันให้อยู่ในมิติข้อมูลเดียวกันเพื่อคำนวณรอบเดียว ผ่านพารามิเตอร์:

    ```protobuf
    # config.pbtxt
    dynamic_batching {
      max_queue_delay_microseconds: 20000  # รอสูงสุด 20 มิลลิวินาที
      preferred_batch_size: [ 4, 8, 16 ]    # มัดรวมกลุ่มขนาดตามที่ GPU ถนัด
    }
    ```

* **เคสการใช้งานจริง:** เมื่อระบบโดนถล่มด้วยภาพเกจพร้อมกัน 8 ภาพ แทนที่ GPU จะรันทีละภาพจนครบ 8 รอบ (ช้าและหน่วง) Triton จะจับภาพทั้ง 8 มัดรวมกันแล้วประมวลผลในรอบเดียว ทำให้ใช้เวลาโดยรวมลดลงเกือบ 8 เท่า

### 5) Sequence Batching for Stateful Models (การจดจำความเชื่อมโยงของเวลา)

* **เจาะลึกการทำงาน:** การผูกความเกี่ยวข้องของเฟรมวิดีโอ (Correlation ID) เพื่อให้โมเดลจำลองจำตรรกะของเฟรมก่อนหน้าได้
* **เคสการใช้งานจริง:** การเฝ้าระวังแก๊สรั่วซึม (Task 4) หรือตรวจจับไฟและควันไฟ (Task 5) ที่ต้องการวัดปริมาณการฟุ้งกระจายของกลุ่มควัน AI จะต้องดูภาพต่อกันเป็นช่วงเวลา 2-3 วินาทีเพื่อความแม่นยำ ป้องกันการเข้าใจผิดจากฝุ่นละอองปกติ

### 6) Backend API for Custom Operations (การรองรับโค้ดตรรกะเฉพาะทาง)

* **เจาะลึกการทำงาน:** เปิดโอกาสให้เขียน Python Script หรือ C++ โค้ดลงไปรันเป็นโมเดลตัวหนึ่งใน Triton (Python Backend) เพื่อแก้ไขและจัดระเบียบภาพก่อนส่งให้ AI
* **เคสการใช้งานจริง:** สคริปต์การคำนวณทางคณิตศาสตร์ระดับสูง เช่น การแปลงผลพิกัดจากโมเดล AI ออกมาเป็นมุมองศาความหนาแน่นเพื่อวิเคราะห์ค่าแรงดันเกจ (Libs Analog Gauge) จะทำงานอยู่ในความเร็วสูงสุดภายใน Triton

### 7) Model Ensembles and Business Logic Scripting (BLS) (ระบบสายพานต่อยอดความฉลาด)

* **เจาะลึกการทำงาน:**
  * **Ensemble:** กำหนดในไฟล์คอนฟิกเพื่อต่อท่อภาพเข้า-ออกโดยไม่มีการคัดลอกหน่วยความจำ (Zero Memory Copy)
  * **BLS:** การควบคุมแบบไดนามิกผ่านสคริปต์ Python ใน Triton ทำให้มีตรรกะการตัดสินใจในขั้นตอนนั้นๆ ได้แบบยืดหยุ่นสูง
* **เคสการใช้งานจริง:** การสร้างโมเดลตัวกลาง `master_router` เพื่อรับ Payload B ตรวจดูประเภทภารกิจ แล้วเลือกว่าจะส่งภาพเกจไปหาโมเดลอ่านค่า หรือส่งภาพแก๊สไปหาโมเดลตรวจจับการฟุ้งกระจายของอากาศ

### 8) Standardized Protocols based on KServe (การเชื่อมต่อระดับมาตรฐานอุตสาหกรรม)

* **เจาะลึกการทำงาน:** สนับสนุน KServe v2 API ทั้งพอร์ต HTTP และ gRPC
* **เคสการใช้งานจริง:** ช่วยสร้างความมั่นใจแก่แผนกไอทีขององค์กรว่า โครงสร้างระบบของเราจะไม่เกิดปัญหาผูกขาดกับผู้พัฒนาเฉพาะราย (Vendor Lock-in) และสามารถนำไปทำงานร่วมกับซอฟต์แวร์ระดับโลกอื่นๆ เช่น Kubeflow หรือ Seldon Core ได้อย่างลื่นไหล

### 9) In-Process C API (การรันแอปพลิเคชันโดยไม่มีเน็ตเวิร์กแลตซี)

* **เจาะลึกการทำงาน:** การดึงไลบรารีแชร์ระดับ C (`libtritonserver.so`) เข้ามาร่วมในตัวซอฟต์แวร์แอปพลิเคชันโดยตรง
* **เคสการใช้งานจริง:** หากต้องการพัฒนากล่องประมวลผลขนาดเล็กติดรถยนต์ลาดตระเวน (Edge Compute Box) เราสามารถถอดพอร์ตการเชื่อมต่อเครือข่ายทั้งหมดออก และให้ซอฟต์แวร์ของหุ่นยนต์เรียกใช้ AI ตรงๆ ผ่านหน่วยความจำกายภาพของตัวบอร์ดได้เลย

### 🔟 Observability & Metrics Export (ระบบตรวจสอบสุขภาพของระบบ)

* **เจาะลึกการทำงาน:** คายข้อมูลรายงานสถิติตามมาตรฐานของโปรแกรมเก็บข้อมูล **Prometheus** ที่พอร์ต `/metrics`
* **เคสการใช้งานจริง:** ทีมวิศวกรระบบไอทีส่วนกลาง (DevOps) สามารถทำกราฟสวยงามบนหน้าจอ **Grafana** เพื่อดูความร้อนการ์ดจอ, ปริมาณการใช้พลังงาน, และความเร็วในการตอบสนองของเซิร์ฟเวอร์ได้แบบเรียลไทม์ 24 ชั่วโมง

## 8. ความปลอดภัยและการปกป้องความเป็นส่วนตัวของข้อมูลระดับรัฐ/องค์กรใหญ่ (Data Privacy & Security Hardening)

ในพื้นที่ปฏิบัติงานที่มีข้อมูลอ่อนไหวระดับความมั่นคงแห่งรัฐหรือความปลอดภัยของโรงงานอุตสาหกรรมพลังงาน (เช่น แท่นขุดเจาะน้ำมัน แหล่งผลิตแก๊สธรรมชาติ) ความปลอดภัยด้านความปลอดภัยทางข้อมูลต้องได้รับการออกแบบให้ไร้จุดโหว่ (Hardened Security):

```
                       [ เครือข่ายภายนอก / หุ่นยนต์ภายนอก ]
                                       │ (HTTPS + Token Validation)
  ┌────────────────────────────────────▼────────────────────────────────────┐
  │ 1. API Security Gateway (Inference Proxy - DMZ เครือข่ายภายนอก)           │
  │    - ยืนยันสิทธิ์ JWT Token / คัดกรองคำร้องขอแปลงภาพ                        │
  └────────────────────────────────────┬────────────────────────────────────┘
                                       │ (เครือข่ายปิดชั้นใน - Internal gRPC Only)
  ┌────────────────────────────────────▼────────────────────────────────────┐
  │ 2. Triton Server AI Engine (รันระบบปิด Offline 100%)                    │
  │    - ประมวลผลบน RAM และ VRAM / ล้างรูปทิ้งหลังส่งผลสำเร็จ                    │
  └─────────────────────────────────────────────────────────────────────────┘
```

* **8.1 100% Air-Gapped & Local Isolation (ระบบตัดขาดเครือข่ายภายนอก):**
    ระบบทั้งหมดสามารถติดตั้งและทำงานได้โดยไม่ต้องเข้าถึงอินเทอร์เน็ตภายนอก (Offline Mode) ปิดโอกาสการแฮกข้อมูลข้ามประเทศหรือการดักฟังข้อมูลกลางทาง ข้อมูลและภาพถ่ายแท่นตรวจวัดน้ำมันและแรงดันแก๊สจะถูกกักอยู่เฉพาะภายในขอบเขตระบบความปลอดภัยขององค์กรเท่านั้น
* **8.2 DMZ Separation & Port Hardening (การแยกโซนเครือข่ายระดับเครือข่าย):**
    ตัวแอปพลิเคชัน Triton จะถูกซ่อนอยู่ในเครือข่ายภายใน (Internal Zone) และห้ามเปิดพอร์ตหาเครือข่ายภายนอกโดยตรง ส่วน Inference Proxy จะถูกจัดอยู่ในโซนคัดกรองข้อมูล (DMZ Layer) ทำหน้าที่เป็นโล่ป้องกันแรงปะทะจากการถล่มยิงเครือข่าย ป้องกันไม่ให้แฮกเกอร์โจมตีไปถึงแกนคำนวณการ์ดจอ
* **8.3 Strict Token-Based Access Controls (การตรวจสอบสิทธิ์หุ่นยนต์รายตัว):**
    เมื่อมีคำขอมาจากหุ่นยนต์ลาดตระเวน ตัว Proxy จะถอดรหัสตรวจสอบข้อมูลเพื่อเช็คคีย์ยืนยัน (JWT Token) หากเป็นอุปกรณ์แปลกปลอมหรือกุญแจความปลอดภัยหมดอายุ ระบบจะทำการตัดการเชื่อมต่อทิ้งทันทีในระดับมิลลิวินาที ป้องกันอุปกรณ์สวมรอย
* **8.4 Zero-Disk Footprint & Volatile Data Purging (การลบภาพในแรมทันที):**
    กระบวนการดาวน์โหลดและประมวลผลรูปภาพทั้งหมดจะดำเนินการอยู่บน **RAM และ VRAM ของการ์ดจอชั่วคราวเท่านั้น** ทันทีที่แปลงผลลัพธ์ไปเป็น Payload C และยิงกลับหาหุ่นยนต์ตรวจเสร็จสิ้น ตัวแปรภาพถ่ายจะถูกเขียนทับและลบออกจากระบบหน่วยความจำโดยอัตโนมัติ ไม่มีการเก็บภาพค้างไว้บน Harddisk ป้องกันความเสี่ยงจากการขโมยข้อมูลทางกายภาพหากเซิร์ฟเวอร์โดนเจาะ
* **8.5 Sandbox Containerization (การกักบริเวณช่องโหว่):**
    การจำกัดขอบเขตการเข้าถึงระดับไฟล์ หากระบบ AI หรือสคริปต์การคำนวณชำรุดเสียหายหรือถูกเจาะ ผู้บุกรุกจะถูกกักตัวไว้เฉพาะในขอบเขตอันจำกัดของตู้ Docker Sandbox เท่านั้น ไม่สามารถข้ามไปควบคุมไฟล์ระบบปฏิบัติการหลัก หรือเครื่องเซิร์ฟเวอร์ระบบไอทีตัวอื่นขององค์กรได้เลย

## 9. การออกแบบระบบเพื่อรองรับหุ่นยนต์และกล้องจำนวนมาก (System Scaling & Concurrency Math)

เพื่อจำลองภาพการทำงานที่หุ่นยนต์หลายสิบตัว (เช่น 30 ตัว) วิ่งตรวจจับอุปกรณ์รอบโรงงานพร้อมๆ กัน เราจะใช้วิธีคำนวณทางวิศวกรรมระบบดังนี้:

### 9.1 สมการการคำนวณภาระงาน (Throughput Math)

* **จำนวนหุ่นยนต์:** 30 ตัว
* **อัตราการยิงข้อมูล:** หุ่นยนต์แต่ละตัวส่งภาพถ่ายมาประมวลผลทุกๆ 3 วินาที
* **คำนวณปริมาณงานต่อวินาที (Load Throughput):**
    $$\text{Requests per Second} = \frac{30 \text{ robots}}{3 \text{ seconds}} = 10 \text{ คำร้องขอต่อวินาที (FPS)}$$
* **ขนาดข้อมูลภาพที่ใช้:** ภาพถ่ายขนาด $1080\text{p} \ (1920 \times 1080)$ ขนาดไฟล์เฉลี่ยประมาณ $2 \text{ MB}$ ต่อภาพ
* **ปริมาณแบนด์วิดท์เครือข่ายขาเข้าที่ต้องแบกรับ:**
    $$\text{Bandwidth Requirement} = 10 \text{ FPS} \times 2 \text{ MB} = 20 \text{ MB/s (หรือ } 160 \text{ Mbps)}$$

### 9.2 กลไกการแก้ปัญหาคิวหน่วงสะสม (System Architecture Solutions)

1. **Horizontal Proxy Scaling (ขยายป้อมยามรับโหลด):**
    ปริมาณ Bandwidth $160\text{ Mbps}$ และการประมวลผลแปลงไฟล์ JPEG ค่อนข้างกินพลังงาน CPU ของ Proxy สูง หาก Proxy ตัวเดียวเริ่มตอบสนองช้าลง เราสามารถตั้งตู้ Proxy เพิ่มเป็น 2 หรือ 3 เครื่องขนานกัน โดยใช้ **NGINX หรือ HAProxy** คอยหมุนกระจายโหลด (Round-Robin Load Balancing)
2. **GPU Dynamic Batching Optimization (การมัดรวมความคุ้มค่าของการ์ดจอ):**
    ด้วยอัตรา $10\text{ คำขอต่อวินาที}$ หมายความว่าในเสี้ยววินาทีหนึ่งจะมีภาพเข้ามาพร้อมกันประมาณ 2-3 ภาพ Triton จะหน่วงรอระยะเวลาสั้นมาก (เช่น $15\text{ มิลลิวินาที}$) เพื่อมัดรวมภาพที่เข้ามาในช่วงนั้นให้กลายเป็น Batch ขนาดใหญ่ยักษ์แล้วโยนตูมเดียวเข้าแกน A6000 ทำให้ GPU รันภาพทั้งหมดเสร็จในรอบการคำนวณเดียว ช่วยประหยัดรอบการทำงานการ์ดจอได้กว่า 70%
3. **Backpressure and Active Queue Management (ระบบตัดโหลดป้องกันเซิร์ฟเวอร์ล่ม):**
    หากเกิดเหตุการณ์ฉุกเฉินที่สัญญาณภาพยิงเข้ามาพร้อมกันสูงเกินขีดจำกัด (เช่น 100 FPS) Triton จะเริ่มเปิดกลไก **Queue Time-to-Live (TTL)** เพื่อระบุว่าหากงานใดค้างอยู่ในคิวรอนานเกินเวลาวิกฤต (เช่น เกิน 5 วินาที) จะคายสิทธิ์ให้หลุดออกจากคิวอัตโนมัติพร้อมแจ้งสถานะ "ระบบงานหนาแน่นเกินพิกัด" เพื่อเปิดทางให้ภาพเหตุการณ์ใหม่ล่าสุดได้ประมวลผลทันที ป้องกันอาการคิวหน่วงสะสมจนค้างยาวทั้งระบบ (System Cascading Failure)

## 10. การวิเคราะห์เปรียบเทียบสถาปัตยกรรม End-to-End gRPC กับ Hybrid (gRPC + JSON)

เพื่อประโยชน์ในการศึกษาเปรียบเทียบสำหรับโครงสร้างระบบปัญญาประดิษฐ์ระดับแนวหน้าของโลก หัวข้อนี้นำเสนอการวิเคราะห์สถาปัตยกรรมการสื่อสารระหว่างการดึงขีดจำกัด gRPC ไปใช้ที่ตัวอุปกรณ์ Edge (หุ่นยนต์) กับการใช้สถาปัตยกรรมแบบผสมผสาน (Hybrid)

```
[สถาปัตยกรรมแบบ Hybrid (ปัจจุบัน)]
หุ่นยนต์ ───► (HTTP/JSON - ง่าย ยืดหยุ่น) ───► Inference Proxy ───► (gRPC/Binary - เร็ว) ───► Triton

[สถาปัตยกรรมแบบ End-to-End gRPC (ทางเลือกความเร็วแสง)]
หุ่นยนต์ ───────────────────────► (gRPC/Binary Tensors ตรง) ───────────────────────► Triton
```

---

### 10.1 สิ่งที่หุ่นยนต์ต้องแบกรับหากเปลี่ยนไปใช้ End-to-End gRPC

หากทีมวิศวกรตัดสินใจตัดชั้น Inference Proxy ออก เพื่อเปิดใช้ gRPC ตั้งแต่อุปกรณ์ Edge จนถึงการ์ดจอโดยตรง สิ่งที่ฝั่งหุ่นยนต์ต้องดำเนินการประกอบด้วย:

1.  **การถือไฟล์ร่วมกัน (Shared Protobuf Schema):** หุ่นยนต์ทุกตัวในระบบจะต้องจัดเก็บไฟล์ข้อตกลงข้อมูลร่วมกัน `.proto` (Protocol Buffers) ซึ่งระบุตัวแปร มิติภาพถ่าย และโครงสร้างเอาต์พุตไว้อย่างเข้มงวด
2.  **การคอมไพล์ไคลเอนต์เฉพาะตัว (Client Stub Compilation):** ซอฟต์แวร์บนหุ่นยนต์ (เช่น โค้ดควบคุมระบบปฏิบัติการ ROS2/C++ หรือ Python) จะต้องถูกคอมไพล์ผ่านคำสั่ง `protoc` เพื่อสร้างคลาสสื่อสารข้อมูลไบนารีที่จับคู่ตรงกับไฟล์ระบบกลาง
3.  **การทำ Serialization บนหน่วยประมวลผลปลายทาง (Edge Serialization):** หุ่นยนต์ต้องแปลงรูปภาพจากกล้องถ่ายรูปให้อยู่ในรูปตัวแปรอะเรย์ไบนารี (Byte Stream) ภายในบอร์ดคำนวณของหุ่นยนต์เอง แล้วยิงผ่านท่อเชื่อมต่อ HTTP/2 เพื่อลดความล่าช้าในเลเยอร์เน็ตเวิร์ก
4.  **การทำ Deserialization เอาต์พุต (Edge Deserialization):** ผลลัพธ์ที่ตอบกลับมาจาก Triton จะเป็นข้อมูล Binary ดิบ หุ่นยนต์ต้องใช้คลาส gRPC ที่คอมไพล์ไว้แปลงข้อมูลกลับมาเป็นตัวแปรธรรมดาในโค้ดเพื่อนำไปสั่งการล้อเลื่อนหรือแขนกล

---

### 10.2 ตารางสรุปการเลือกใช้งาน (Decision Criteria Matrix)

| ปัจจัยการประเมิน | สถาปัตยกรรมแบบผสม (Hybrid: JSON + gRPC) | สถาปัตยกรรม gRPC เต็มพิกัด (End-to-End) |
| :--- | :--- | :--- |
| **การบำรุงรักษาระบบ (Maintenance)** | **ดีเลิศ:** หากมีการเปลี่ยนแปลงสเปค หรือ อัปเกรดโมเดล AI ใหม่ สามารถจัดการที่ตัว Proxy ได้ทันที หุ่นยนต์หน้างานไม่ต้องแก้ไขโค้ดใดๆ | **ยากขึ้น:** หากมีการปรับสเปคตัวแปรในไฟล์ `.proto` วิศวกรต้องทำการแฟลชเฟิร์มแวร์และอัปเกรดซอฟต์แวร์หุ่นยนต์ทุกตัวพร้อมกัน |
| **ความเร็วในการตอบสนอง (Latency)** | **ต่ำ (10-50 ms):** มีหน่วงเวลาเพิ่มขึ้นเล็กน้อยในส่วนการแปลงฟอร์แมตและการจัดคิวที่ Proxy (CPU) | **ต่ำที่สุดขีด (1-5 ms):** ส่งตรงผ่านท่อ HTTP/2 ไบนารี ไม่ผ่านตัวกลางการคำนวณ CPU ใดๆ |
| **การใช้ปริมาณเครือข่าย (Bandwidth)** | **ปานกลาง:** ข้อมูลมีการส่งในรูปแบบ JSON ซึ่งมีขนาดหัวจดหมายใหญ่กว่า หรือต้องส่งรูปผ่านการดึง URL | **ประหยัดสูงสุด:** บีบอัดตัวแปรให้อยู่ในรูปแบบ Byte ดิบที่มีขนาดเล็กที่สุด ส่งผลต่อ Wi-Fi หน้างานน้อยที่สุด |
| **ความเสถียรบนโครงข่ายไร้สาย (Wi-Fi Stability)** | **ดี:** การเชื่อมต่อ HTTP REST ทำงานแบบไร้สถานะ (Stateless) หากคลื่น Wi-Fi หลุดชั่วขณะ สามารถสลับยิงใหม่ได้ง่าย | **ต้องตั้งค่าเพิ่มเติม:** การเชื่อมต่อ gRPC บน HTTP/2 อาศัยท่อสตรีมระยะยาว (Persistent Connection) หากสัญญาณ Wi-Fi หน้างานขาดหาย ต้องมีตรรกะ Retry Connection ที่ทนทานสูง |

---

### 10.3 สรุปคำแนะนำเชิงระบบ (Architectural Recommendation)

*   **ควรเลือกใช้ End-to-End gRPC เมื่อ:** เป็นระบบหุ่นยนต์ตอบสนองไววิกฤตสูง (เช่น แขนกลคัดแยกสิ่งของความเร็วสูงบนสายพาน, โดรนไร้คนขับหลบหลีกสิ่งกีดขวาง) ที่แลตซีต้องต่ำกว่า 10 มิลลิวินาที และเน้นใช้งานในสภาวะที่โครงข่ายเครือข่ายมีความเสถียรสูงมาก (เช่น การต่อสาย LAN หรือ เครือข่าย 5G Private Network ประจำโรงงาน)
*   **ควรเลือกใช้แบบผสม Hybrid (ที่เราใช้ในปัจจุบัน) เมื่อ:** เป็นงานตรวจตราอุตสาหกรรมทั่วไป (เช่น หุ่นยนต์ตรวจเกจวันละ 3 รอบ, งานเฝ้าระวังควันไฟที่อัปเดตผลวินาทีละครั้ง) ที่เน้นความทนทานและการดูแลรักษาง่ายเป็นสำคัญ เพื่อเปิดโอกาสให้ทีมพัฒนาโมเดล AI สลับเปลี่ยนเวอร์ชันโมเดลได้แบบเรียลไทม์โดยไม่มีความเสี่ยงต่อการหยุดชะงักของฝูงหุ่นยนต์ (Robot Fleet) หน้างาน

---

## 11. สถาปัตยกรรมทางเลือก: เมื่อย้าย Inference Proxy ไปรันบนตัวหุ่นยนต์ (Edge-Side Proxy Architecture)

หากสถาปนิกตัดสินใจย้าย **AI Inference Proxy** จากระบบคลาวด์/เซิร์ฟเวอร์หลังบ้าน ไปรันอยู่บนตัวหุ่นยนต์ตรวจการณ์โดยตรง (เช่น รันในรูปแบบ Docker Container บนหน่วยประมวลผลของหุ่นยนต์ เช่น NVIDIA Jetson Orin หรือ Intel NUC) สถาปัตยกรรมจะเปลี่ยนรูปไปดังนี้:

```
[โครงสร้าง Edge-Side Proxy (Proxy บนตัวหุ่นยนต์)]
┌───────────────────────────── หุ่นยนต์ (Robot Hardware Edge) ─────────────────────────────┐
│ [แอปพลิเคชันหุ่นยนต์ / ROS2] ───► (HTTP/JSON - localhost) ───► [Inference Proxy (แรมหุ่น)] │
└───────────────────────────────────────────────────────────────┬──────────────────────────┘
                                                                │
                                                                │ gRPC / Binary Tensors ดิบ
                                                                │ ยิงผ่านเครือข่ายไร้สาย Wi-Fi/5G
                                                                ▼
                                                     [ NVIDIA Triton Server ]
                                                       (ประมวลผลบน GPU เซิร์ฟเวอร์กลาง)
```

---

### 11.1 ข้อดีของการย้าย Proxy ไปรันบนตัวหุ่นยนต์ (Architectural Advantages)

1.  **กำจัดคอขวดแบนด์วิดท์ Wi-Fi หน้างานอย่างมีนัยสำคัญ (Massive Bandwidth Saving):**
    *   **ก่อนหน้านี้:** หุ่นยนต์ต้องอัปโหลดภาพ JPEG ขนาดเต็ม (เช่น ภาพความละเอียด 4K หรือ 12MP ขนาด 5MB-10MB) ขึ้นไปยังเซิร์ฟเวอร์เพื่อให้ Proxy ดาวน์โหลด
    *   **เมื่อ Proxy อยู่บนหุ่นยนต์:** ตัว Proxy บนหุ่นยนต์สามารถแปลงไฟล์ภาพดิบ ทำการลดขนาดภาพ (Resizing เหลือ 640x640) และสกัดเฉพาะพื้นที่สนใจ (Cropping) ให้กลายเป็น Binary Tensors ขนาดเล็ก (เช่น ต่ำกว่า 1MB) *ภายในหุ่นยนต์เองก่อน* แล้วจึงส่งสัญญาณเฉพาะ Tensors ดิบที่พร้อมประมวลผลข้าม Wi-Fi ส่งผลให้ประหยัดแบนด์วิดท์เครือข่ายไร้สายในโรงงานปฏิบัติงานไปได้กว่า 80-90%
2.  **ขจัดคอขวดฝั่งเซิร์ฟเวอร์ I/O (Server-Side CPU Relief):**
    *   เซิร์ฟเวอร์ส่วนกลางจะไม่ต้องแบกรับภาระงานดาวน์โหลดไฟล์ภาพจาก S3/Cloud Storage หรือต้องมารันโค้ด CPU เพื่อถอดรหัสรูปภาพ JPEG อีกต่อไป เพราะหน้าที่การประมวลผลเตรียมภาพ (Preprocessing) ถูกกระจายไปให้ CPU ของหุ่นยนต์แต่ละตัวช่วยกันทำ (Distributed Computing)
3.  **โค้ดแอปพลิเคชันหุ่นยนต์ยังคงเรียบง่าย (Maintains Thin Client App Code):**
    *   ซอฟต์แวร์หลักของหุ่นยนต์ (เช่น โค้ด ROS2/Python) ไม่จำเป็นต้องรู้วิธีการคุย gRPC หรือจัดเตรียมมิติตัวเลข Tensors ซับซ้อนด้วยตัวเอง มันยังคงส่งคำสั่งควบคุม HTTP/JSON แบบง่ายไปหา `localhost:5000` (Inference Proxy ท้องถิ่นบนหุ่น) ได้เช่นเดิม ทำให้ง่ายต่อการพัฒนาระบบควบคุมหุ่นยนต์

---

### 11.2 ข้อเสียและข้อพิจารณาที่ต้องระวัง (Architectural Trade-offs)

1.  **ความซับซ้อนในการอัปเดตและดูแลรักษา (Edge Fleet Maintenance Overhead):**
    *   หากมีการปรับเปลี่ยนพารามิเตอร์การจัดขนาดภาพ (Preprocessing Specs) หรือเปลี่ยนเวอร์ชัน API เอาต์พุต (Payload C) วิศวกรจะต้องไล่อัปเดตอิมเมจ Docker Container ของ Proxy บนตัวหุ่นยนต์ตรวจการณ์**ทุกตัว**ที่วิ่งอยู่ในสนามปฏิบัติงาน ซึ่งมีความเสี่ยงและยากกว่าการปรับแต่งโค้ด Proxy จุดเดียวบนเซิร์ฟเวอร์กลาง
2.  **การสิ้นเปลืองแบตเตอรี่และพลังงานของหุ่นยนต์ (Edge Energy Consumption):**
    *   กระบวนการถอดรหัสภาพและเตรียมอะเรย์คณิตศาสตร์ใช้ทรัพยากร CPU/RAM ของบอร์ดประมวลผลบนตัวหุ่นยนต์ ซึ่งอาจเพิ่มอัตราการใช้พลังงานส่งผลให้ชั่วโมงการวิ่งปฏิบัติงานต่อการชาร์จแบตเตอรี่ 1 ครั้งของหุ่นยนต์ลดลงเล็กน้อย
3.  **ข้อจำกัดเรื่องการกระจายน้ำหนักโหลดส่วนกลาง (Loss of Centralized Control):**
    *   หากเกิดกรณีเซิร์ฟเวอร์ Triton ตัวหลักเกิดปัญหาคอขวด ตัว Proxy บนหุ่นยนต์จะไม่สามารถช่วยกระจายคำขอไปยังเซิร์ฟเวอร์สำรองเครื่องอื่นๆ ได้ดีเท่ากับระบบ API Gateway ส่วนกลาง (Centralized Proxy Layer) ที่สามารถทำหน้าที่ Load Balancing ได้สมบูรณ์แบบกว่า
